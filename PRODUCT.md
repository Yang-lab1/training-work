# Product

## Register

product

## Users

Primary user: a job seeker moving from industrial design and AI-related graduate study into AI product, AI hardware, AI application, UX research, TPM, and solution-oriented roles.

The user works inside a practical interview-prep loop: upload CV and job data, pick a target role, prepare company and role material, practice answers, run mock interviews, review real interviews, and preserve progress across sessions.

## Product Purpose

Interview OS is a local-first interview training workspace. It turns a job database, CV text, training records, company material, and provider feedback into a repeatable preparation loop.

Success means the user can choose a role, prepare quickly, practice in a realistic interview room, get transcript and AI feedback, review real interviews, export/import data, and always understand whether the system is using a real provider or mock fallback.

## Brand Personality

Calm, precise, capable.

The interface should feel like a focused training cockpit rather than a content site: restrained, direct, and trustworthy. It should reduce friction, avoid decorative noise, and make the user feel oriented before each action.

## Anti-references

- Not a marketing landing page.
- Not a dashboard of decorative cards.
- Not a form-heavy recorder.
- Not a chat app.
- Not a long instruction manual inside the interview room.
- Not a mock result pretending to be a real model.
- Not a UI that asks the user to manually recreate job data already present in `job.xlsx`.

## Design Principles

1. Make the current task obvious: one screen, one next action, short state words.
2. Keep interview mode immersive: meeting-room layout first, supporting material hidden until needed.
3. Preserve trust: provider state, fallback state, and missing keys must be visible without exposing secrets.
4. Keep data durable: selected job, normalized role data, records, packs, transcripts, feedback, and imports must survive refresh.
5. Prefer quiet competence over cleverness: familiar controls, stable layout, clear error recovery.

## Accessibility & Inclusion

Target WCAG AA contrast and keyboard-reachable controls. Mobile must avoid horizontal overflow, cramped controls, and clipped text. Reduced-motion users should still get state changes without decorative animation. Error and empty states should be short, actionable, and not depend on color alone.
